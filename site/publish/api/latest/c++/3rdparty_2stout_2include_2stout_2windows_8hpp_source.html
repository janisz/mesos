79 // This code un-defines the global `GetMessage` macro defined by the Windows
80 // headers, and replaces it with an inline function that is equivalent.
81 //
82 // There are two reasons for doing this. The first is because this macro
83 // interferes with `google::protobufs::Reflection::GetMessage`. Replacing the
84 // `GetMessage` macro with an inline function allows people calling the
85 // `GetMessage` macro to carry on doing so with no code changes, but it will
86 // also allow us to use `google::protobufs::Reflection::GetMessage` without
87 // interference from the macro.
88 //
89 // The second is because we don't want to obliterate the `GetMessage` macro for
90 // people who include this header, either on purpose, or incidentally as part
91 // of some other Mesos header. The effect is that our need to call protobuf's
92 // `GetMessage` function has no deleterious effect on customers of this API.
93 //
94 // NOTE: the Windows headers also don't use define-once semantics for the
95 // `GetMessage` macro. In particular, this means that every time you include
96 // `Winuser.h` and a `GetMessage` macro isn't defined, the Windows headers will
97 // redefine it for you. The impact of this is that you should re-un-define the
98 // macro every time you include `Windows.h`; since we should be including
99 // `Windows.h` only from this file, we un-define it just after we include
100 // `Windows.h`.

103  LPMSG lpMsg,

105  UINT wMsgFilterMin,
106  UINT wMsgFilterMax)

108  return GetMessage(lpMsg, hWnd, wMsgFilterMin, wMsgFilterMax);

215 // Permissions API. (cf. MESOS-3176 to track ongoing permissions work.)
216 //
217 // We are currently able to emulate a subset of the POSIX permissions model
218 // with the Windows model:
219 // [x] User write permissions.
220 // [x] User read permissions.
221 // [ ] User execute permissions.
222 // [ ] Group permissions of any sort.
223 // [ ] Other permissions of any sort.
224 // [x] Flags to control "fallback" behavior (e.g., we might choose
225 // to fall back to user readability when the user passes the
226 // group readability flag in, since we currently do not support
227 // group readability).
228 //
229 //
230 // Rationale:
231 // Windows currently implements two permissions models: (1) an extremely
232 // primitive permission model it largely inherited from DOS, and (2) the Access
233 // Control List (ACL) API. Because there is no trivial way to map the classic
234 // POSIX model into the ACL model, we have implemented POSIX-style permissions
235 // in terms of the DOS model. The result is the permissions limitations above.
236 //
237 //
238 // Flag implementation:
239 // Flags fall into the following two categories.
240 // (1) Flags which exist in both permission models, but which have
241 // different names (e.g., `S_IRUSR` in POSIX is called `_S_IREAD` on
242 // Windows). In this case, we define the POSIX name to be the Windows
243 // value (e.g., we define `S_IRUSR` to have the same value as `_S_IREAD`),
244 // so that we can pass the POSIX name into Windows functions like
245 // `_open`.
246 // (2) Flags which exist only on POSIX (e.g., `S_IXUSR`). Here we
247 // define the POSIX name to be the value given in the glibc
248 // documentation[1], shifted left by 16 bits (since `mode_t`
249 // is unsigned short on POSIX and `int` on Windows). We give these
250 // flags glibc values to stay consistent, and so that existing
251 // calls to functions like `open` do not break when they try to
252 // use a flag that doesn't exist on Windows. But, of course,
253 // these flags do not affect the execution of these functions.
254 //
255 //
256 // Flag strictness:
257 // Because the current implementation does not directly support setting or
258 // getting group or other permission bits on the Windows platform, there is a
259 // question of what we should fall back to when these flags are passed in to
260 // Stout methods.
261 //
262 // TODO(hausdorff): Investigate permissions mappings.
263 // We force "strictness" of the permission flag semantics:
264 // * The group permissions flags will not fall back to anything, and will be
265 // completely ignored.
266 // * Other permissions: Same as above, but with other permissions.
267 //
268 //
269 // Execute permissions:
270 // Because DOS has no notion of "execute permissions", we define execute
271 // permissions to be read permissions. This is not ideal, but it is closest to
272 // being accurate.
273 //
274 //
275 // [1] http://www.delorie.com/gnu/docs/glibc/libc_288.html
276

368 inline auto mkstemp(char* path) ->
369 decltype(_mktemp_s(path, strlen(path) + 1))
370 {
371  // NOTE: in the POSIX spec, `mkstemp` will generate a random filename from
372  // the `path` template, `open` that filename, and return the resulting file
373  // descriptor. On Windows, `_mktemp_s` will actually only generate the path,
374  // so here we actually have to call `open` ourselves to get a file descriptor
375  // we can return as a result.
376  if (_mktemp_s(path, strlen(path) + 1) != 0) {
377  return -1;
378  }
379 
380  // NOTE: We open the file with read / write access for the given user, an
381  // attempt to match POSIX's specification of `mkstemp`. We use `_S_IREAD` and
382  // `_S_IWRITE` here instead of the POSIX equivalents. On Windows the file is
383  // is not present, we use `_O_CREAT` option when opening the file.
384  return _open(path, _O_CREAT, _S_IREAD | _S_IWRITE);
385 }
