26 // The dispatch mechanism enables you to "schedule" a method to get
27 // invoked on a process. The result of that method invocation is
28 // accessible via the future that is returned by the dispatch method
29 // (note, however, that it might not be the _same_ future as the one
30 // returned from the method, if the method even returns a future, see
31 // below). Assuming some class 'Fibonacci' has a (visible) method
32 // named 'compute' that takes an integer, N (and returns the Nth
33 // fibonacci number) you might use dispatch like so:
34 //
35 // PID<Fibonacci> pid = spawn(new Fibonacci(), true); // Use the GC.
36 // Future<int> f = dispatch(pid, &Fibonacci::compute, 10);
37 //
38 // Because the pid argument is "typed" we can ensure that methods are
39 // only invoked on processes that are actually of that type. Providing
40 // this mechanism for varying numbers of function types and arguments
41 // requires support for variadic templates, slated to be released in
42 // C++11. Until then, we use the Boost preprocessor macros to
43 // accomplish the same thing (all be it less cleanly). See below for
44 // those definitions.
45 //
46 // Dispatching is done via a level of indirection. The dispatch
47 // routine itself creates a promise that is passed as an argument to a
48 // partially applied 'dispatcher' function (defined below). The
49 // dispatcher routines get passed to the actual process via an
50 // internal routine called, not suprisingly, 'dispatch', defined
51 // below:
52 
53 namespace internal {

55 // The internal dispatch routine schedules a function to get invoked
56 // within the context of the process associated with the specified pid
57 // (first argument), unless that process is no longer valid. Note that
58 // this routine does not expect anything in particular about the
59 // specified function (second argument). The semantics are simple: the
60 // function gets applied/invoked with the process as its first
61 // argument. Currently we wrap the function in a shared_ptr but this
62 // will probably change in the future to unique_ptr (or a variant).
63 void dispatch(
64  const UPID& pid,
65  const std::shared_ptr<std::function<void(ProcessBase*)>>& f,
66  const Option<const std::type_info*>& functionType = None());

79 template <typename T>
80 void dispatch(const PID<T>& pid, void (T::*method)())
81 {
82  std::shared_ptr<std::function<void(ProcessBase*)>> f(
83  new std::function<void(ProcessBase*)>(
84  [=](ProcessBase* process) {
85  assert(process != nullptr);
86  T* t = dynamic_cast<T*>(process);
87  assert(t != nullptr);
88  (t->*method)();
89  }));
90 
91  internal::dispatch(pid, f, &typeid(method));
92 }
