40 // This isolator uses the cgroups devices subsystem to control
41 // access to Nvidia GPUs. Since this is the very first device
42 // isolator, it currently contains generic device isolation
43 // logic that needs to be pulled up into a generic device
44 // isolator.
45 //
46 // GPUs are allocated to containers in an arbitrary fashion.
47 // For example, if a container requires 2 GPUs, we will
48 // arbitrarily choose 2 from the GPUs that are available.
49 // This may not behave well if tasks within an executor use
50 // GPUs since we cannot identify which task are using which
51 // GPUs (i.e. when a task terminates, we may remove a GPU
52 // that is still being used by a different task!).
53 //
54 // Note that this isolator is not responsible for ensuring
55 // that the necessary Nvidia libraries are visible in the
56 // container. If filesystem isolation is not enabled, this
57 // means that the container can simply use the libraries
58 // available on the host. When filesystem isolation is
59 // enabled, it is the responsibility of the operator /
60 // application developer to ensure that the necessary
61 // libraries are visible to the container (note that they
62 // must be version compatible with the kernel driver on
63 // the host).
64 //
65 // TODO(klueska): To better support containers with a
66 // provisioned filesystem, we will need to add a mechanism
67 // for operators to inject the libraries as a volume into
68 // containers that require GPU access.
69 //
70 // TODO(klueska): If multiple containerizers are enabled,
71 // they need to co-ordinate their allocation of GPUs.
72 //
73 // TODO(klueska): Move generic device isolation logic
74 // out into its own component.
